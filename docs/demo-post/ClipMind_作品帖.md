# 【学习工作】ClipMind — AI 剪贴板知识管家

## 一、Demo 简介

**产品名称：** ClipMind — AI 剪贴板知识管家

**一句话介绍：** 把 macOS 系统剪贴板升级为会自动分类、总结、搜索和复用的 AI 信息库。

**产品形态：** macOS 原生桌面 App（Swift / SwiftUI），菜单栏常驻，复制即理解。

### 面向谁

程序员、学生、文字工作者、产品经理、内容创作者、研究人员——所有经常在 Mac 上复制资料、整理信息的人。

### 解决的核心问题

macOS 用户每天会复制大量信息：代码、链接、报错、文案、会议内容。但普通剪贴板只是一个临时容器，存在四大痛点：

| 痛点 | 具体表现 |
|------|----------|
| 信息被覆盖 | 复制新内容会顶掉旧内容，重要信息稍纵即逝 |
| 无法被理解 | 普通剪贴板只按时间排列，看不懂内容含义 |
| 需手动整理 | 复制后的长文本要自己总结、翻译、改写、归类 |
| 隐私无保护 | 密码、Token、验证码等敏感信息明文留存，缺少提醒 |

### 核心功能

- **自动分类**：复制内容自动识别为代码、链接、文章、报错、待办、会议纪要等 12+ 种类型，打上类型标签
- **语义搜索**：用自然语言搜索剪贴历史，比如「上次那个窗口管理的报错」「关于性能优化的文章」，不必记原词
- **一键处理**：对任意剪贴内容一键执行总结、翻译、改写、提取待办 4 种 AI 处理
- **隐私保护**：本地加密存储 + 6 种敏感模式识别 + 应用黑名单 + 30 天自动清理

### 技术栈

| 层级 | 技术选型 |
|------|----------|
| 客户端 | Swift / SwiftUI |
| 本地 AI | CoreML（嵌入模型 all-MiniLM-L6-v2，22 MB，推理 < 100 ms） |
| 加密 | AES-256-GCM |
| 云端 LLM | 可选，仅用于一键处理（用户自配 API Key） |

## 二、Demo 创作思路

### 灵感来源

在用 Mac 开发、查资料、写文档时，我会临时复制很多有用信息：一段 Xcode 报错、一段接口文档、一条产品需求、一个参考链接。当时觉得「等下会用」，但过一会儿就被新的复制内容覆盖了。真正需要时，只能重新翻网页、翻聊天记录、翻搜索历史，非常浪费时间。

普通剪贴板工具只能保存历史，不能理解内容，更不能帮用户整理和复用。我想把剪贴板从「临时复制工具」升级为「AI 个人知识管家」。

### 设计理念：本地优先（Local First）

ClipMind 的核心设计理念是**本地优先**——数据不出本机。

- 嵌入模型（all-MiniLM-L6-v2）转换成 CoreML 格式后完全本地推理，分类和搜索不依赖云端
- 剪贴内容用 AES-256-GCM 加密存储在本地数据库
- 云端 LLM 仅用于一键处理（总结/翻译/改写/待办），且由用户自行配置 API Key，可选可关

### 架构决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 交互形态 | 菜单栏常驻 + popover 快速预览 + 主窗口完整管理 | 复制是高频操作，入口要轻；管理是低频操作，功能要全 |
| AI 策略 | 本地嵌入模型支持分类和搜索，云端 LLM 仅用于一键处理 | 分类和搜索是高频调用，必须本地秒级响应；一键处理是低频按需，可走云端 |
| 隐私设计 | 加密存储 + 敏感识别 + 黑名单 + 自动清理 | 剪贴板常出现敏感内容，隐私保护是底线 |

### 隐私设计细节

- **AES-256 加密存储**：所有剪贴内容加密后写入本地数据库
- **6 种敏感模式识别**：密码、Token、验证码、身份证号、手机号、银行卡号，识别后不存储并提醒用户
- **应用黑名单**：密码管理器等指定应用复制时不记录
- **30 天自动清理**：过期内容自动删除，减少留存风险

### 用户体验原则

- **复制即理解**：用户复制后无需任何操作，内容已自动分类打标
- **搜索说人话**：用自然语言描述意图即可找到，不必记住原词
- **处理一键完成**：总结、翻译、改写、提取待办，点一下就好

### 开发方法

采用 **TDD（测试驱动开发）**+ **5 Phase 渐进式开发** + **CI 持续集成**：

- 先写测试再写实现，确保每个功能有测试覆盖
- 5 个 Phase 串行推进：基础设施 → 核心 AI → 一键处理 → 隐私与设置 → Web + Demo 帖
- GitHub Actions CI 每次提交自动跑测试和 SwiftLint，保证代码质量

## 四、TRAE 实践过程

### 开发工具

- **IDE：** TRAE IDE
- **开发流程：** 使用 `dd-feature-development-workflow`（设计规范优先的工作流）

### 开发流程说明

整个项目采用「设计规范优先」的工作流：先写设计规范文档（明确 25 条验收标准 AC），再写测试用例表（69 个测试用例），然后拆解实施计划（35 个细粒度任务），最后逐任务用 TDD 方式实现。每个任务在一个 TRAE Session 内完成，完成后有三子代理并行检查机制做代码审查。

### TRAE 价值

- **设计规范优先的工作流**：先想清楚再做，确保代码质量和功能完整性，避免返工
- **三子代理并行检查机制**：多个子代理从不同维度并行审查代码，保证审查全面性
- **CI 持续集成**：每次提交自动跑测试和 SwiftLint，确保质量门禁
- **TDD 开发方法**：测试先行，35 个任务每个都有对应测试，测试覆盖率高

### 开发心得

用 TRAE IDE 开发 ClipMind 的最大感受是「流程可控」。设计规范优先的工作流让我不会一上来就写代码，而是先把验收标准、测试用例、任务拆解想清楚。每个 Session 聚焦单一任务，上下文清晰，三子代理审查机制能在早期发现问题。整个过程像搭积木——每个任务是一块积木，搭完 35 块，产品就成型了。
