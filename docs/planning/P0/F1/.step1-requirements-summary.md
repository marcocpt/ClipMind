# 步骤 1 需求摘要（F1 · ClipMind 实施计划）

> 创建时间：2026-07-12 | 基于 grill 拷问确认

## 1. 用户问题和业务目标

**问题**：ClipMind 设计规范 v1.3 和测试用例表 v1.1 已确认，但缺少将设计规范转化为可执行任务的实施计划。需要在初赛截止（2026-07-15）前，将 5 个 Phase、25 条 AC、69 个测试用例拆解为可执行任务，供 TRAE IDE 逐任务执行。

**目标**：产出一份细粒度实施计划，每个任务可由 1 个 TRAE Session 完成，包含工时估算、依赖关系、AC/测试用例映射、验证方式，并定义 MVP 兜底路径。

## 2. 成功标准

- 实施计划覆盖设计规范全部 25 条 AC 和 69 个测试用例
- 每个任务有明确的输入、输出、依赖、AC 映射、测试用例映射、验证方式
- 每个任务有工时估算
- 定义 MVP 兜底路径（时间不足时的最小任务集）
- 通过 3 子代理并行审查

## 3. 范围边界

**必须做**：
- 将 Phase 0-4 拆解为细粒度任务（按文件/模块）
- 每个任务映射 AC 和测试用例
- 标注任务依赖与执行顺序
- 包含工时估算
- 定义 MVP 兜底路径
- 包含环境搭建任务（Xcode 项目、SwiftLint、依赖库）
- 包含测试数据集准备任务

**不做**：
- 不修改设计规范和测试用例表的内容
- 不编写代码（实施计划是文档，不是代码）
- 不覆盖 F2.x 及之后阶段

## 4. 用户流程

实施计划的使用流程：
1. 开发者按实施计划顺序执行任务
2. 每个任务在 1 个 TRAE Session 内完成
3. 每个任务采用 TDD：先写测试，再写实现
4. 每次 git 提交前执行 SwiftLint 检查
5. 任务完成后更新状态标注（⏸️→🔄→✅）
6. 若时间不足，切换到 MVP 兜底路径

## 5. 数据和接口

### 输入
- 设计规范 v1.3（`docs/planning/P0/F1/F1_ClipMind_设计规范.md`）
- 测试用例表 v1.1（`docs/planning/P0/F1/F1_ClipMind_测试用例表.md`）
- 视觉原型（`docs/planning/P0/F1/F1_ClipMind_视觉原型.html`）

### 输出
- 实施计划文档：`docs/planning/P0/F1/F1_ClipMind_实施计划.md`

### 文档结构
1. 背景与目标（引用设计规范）
2. 实施总览（Phase 依赖图、里程碑、MVP 兜底路径）
3. 环境与工具要求（Xcode、SwiftLint、CoreML、依赖库）
4. 任务分解（按 Phase 0-4 拆分，每个任务含：编号、描述、依赖、输入、输出、AC 映射、测试用例映射、验证方式、工时估算、Session 编号）
5. 任务依赖与执行顺序（拓扑排序、关键路径标注）
6. MVP 兜底路径（最小任务集）
7. 风险与应对（引用设计规范 12.1-12.3 + 新增实施风险）
8. 验收与交付清单（AC 覆盖矩阵、交付物清单、DoD）
9. 执行跟踪机制（任务状态标注规则）

## 6. 兼容和迁移

- 新项目，无历史代码迁移
- 实施计划不修改已有文档
- 实施计划版本与设计规范版本绑定：`vX.Y（基于设计规范 v1.3）`

## 7. 验收标准 AC

实施计划本身的验收标准：
- AC-PLAN-01：覆盖设计规范全部 25 条 AC
- AC-PLAN-02：覆盖测试用例表全部 69 个测试用例
- AC-PLAN-03：每个任务有明确的依赖、输入、输出、验证方式
- AC-PLAN-04：每个任务有工时估算
- AC-PLAN-05：定义 MVP 兜底路径
- AC-PLAN-06：包含环境搭建和测试数据集准备任务
- AC-PLAN-07：通过 3 子代理并行审查

## 8. 阶段拆分

### 确认的执行顺序（串行）
Phase 0 → Phase 1 → Phase 2 → Phase 3 → Phase 4

### MVP 兜底路径
- Phase 0：完整完成
- Phase 1：完整完成
- Phase 2：完整完成
- Phase 3：简化为仅敏感识别 + 应用黑名单（跳过首启引导和完整设置面板）
- Phase 4：完整完成（Web 预览页 + Demo 帖）

不涉及（实施计划是文档，无 UI 交互）。

- 优先级：P0
- 文档路径：`docs/planning/P0/F1/F1_ClipMind_实施计划.md`

## 11. 关键决策汇总

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 任务粒度 | 细粒度（按文件/模块） | 每个 Session 目标明确，便于 TRAE 执行和审查 |
| Phase 执行顺序 | 串行：P0→P1→P2→P3→P4 | 核心功能优先，Phase 2 依赖 Phase 1 无等待 |
| Session 映射 | 每任务 1 Session | 每个 Session 聚焦单一任务，上下文清晰 |
| 嵌入模型 | all-MiniLM-L6-v2 + CoreML | 设计规范推荐，模型小、推理快 |
| 工时估算 | 包含 | 便于进度管控，在截止前可调整优先级 |
| 开发方法 | TDD（测试先行） | 符合设计规范测试策略，AC 验证明确 |
| MVP 兜底 | P0+P1+P2+简化 P3+P4 | 核心功能完整，隐私体验简化 |
