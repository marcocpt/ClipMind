## 1. 测试框架说明

| 框架 | 用途 | 覆盖范围 | 测试模式 |
|------|------|---------|---------|
| XCTest | 单元测试与集成测试，验证业务逻辑 | 白名单检查、长度检查、敏感检查、文件名生成、冲突处理、路径格式化、配置持久化、状态机转换、CaptureEvent 注入、并发场景 | CaptureEvent 注入模式（D8）：构造不可变 `CaptureEvent`（含 bundleId、content、sensitiveResult、配置快照）直接注入 F2.1 钩子或 PasteboardWatcher |
| XCUITest | UI 测试，验证端到端 UI 交互 | 配置面板交互、白名单 App 管理、二次确认弹窗、路径格式切换、配置持久化 | 真实 ClipMind App 启动，模拟用户操作 |
| 手动测试 | 验证无法自动化的场景 | 真实剪贴板来源 App 识别（OS 边界）、真实文件系统权限、真实磁盘满场景 | 真实 Safari/Notes/Xcode 复制时 Bundle ID 识别 |

**测试集存放**：
- 单元测试：`ClipMindTests/`
- UI 测试：`ClipMindUITests/`
- 测试 Fixtures：`ClipMindTests/Fixtures/`
- CaptureEvent Fixtures：`ClipMindTests/Fixtures/CaptureEventFixtures.swift`

**异步等待策略**（D17 落地）：
- ❌ 禁止使用 `sleep 3` 等固定 sleep 等待异步结果
- ✅ 改为轮询目标条件 + 超时 3s + 条件满足即继续
- ✅ 轮询工具：`PollingHelper.waitUntil(condition:timeout:)`（10ms 轮询间隔，3s 超时）

## 8. 测试策略

### 8.1 单元测试与集成测试策略（XCTest，D8 落地）

- **测试目标**：验证设计文档 v1.1 中各模块的职责边界与内部逻辑
- **测试组织**：按模块分文件（如 `WhitelistCheckerTests.swift`、`FileNameGeneratorTests.swift`、`CaptureEventFixtures.swift` 等）
- **CaptureEvent 注入模式**（D8）：构造不可变 `CaptureEvent`（含 bundleId、content、sensitiveResult、f2xConfigSnapshot）直接注入 F2.1 钩子或 PasteboardWatcher，验证文件名、文件内容、剪贴板输出、历史入库调用
- **测试 Fixtures**：测试数据存放在 `ClipMindTests/Fixtures/`，CaptureEvent fixture 存放在 `ClipMindTests/Fixtures/CaptureEventFixtures.swift`
- **Mock 策略**：仅 mock 外部依赖（F1.x 敏感识别模块、F1.x 应用识别模块、F1.x 加密存储模块、F1.x 黑名单服务、文件系统、剪贴板），不 mock 被测对象本身
- **临时目录**：文件写入测试使用临时目录（`NSTemporaryDirectory()`），不依赖真实文件系统状态
- **异步等待**（D17）：禁止使用 `sleep 3`，改为 `PollingHelper.waitUntil(condition:timeout:)`（10ms 轮询间隔，3s 超时）
- **覆盖率目标**：每个新增模块的单元测试覆盖率 ≥ 90%

### 8.2 UI 测试策略（XCUITest，D19 落地）

- **测试目标**：验证用户可见的端到端 UI 交互，覆盖 UI 可观测性矩阵中的所有 UI AC
- **测试组织**：按 AC 分文件（如 `AutoSaveSettingsUITests.swift`）
- **覆盖范围**（D19 最小化）：只覆盖 AC-07（配置生效）、AC-09（异常提示 UI）、AC-15（白名单管理 UI）、AC-16（配置持久化）
- **真实入口**：使用真实 ClipMind App 启动，模拟用户操作
- **断言策略**：配置持久化断言、弹窗断言、白名单列表断言
- **证据存储**：截图与录屏存放在 `docs/planning/P1/F2.1/screenshots/` 与 `recordings/`
- **TC-UI-05（v1.3 新增）**：AC-07 配置面板验证 TextField 存在（`lengthThresholdField` / `fileNameLengthField`），替换原 Stepper 验证（决策 C1）。已在 `AutoSaveSettingsUITests.testAC07AllConfigControlsExist` 添加存在性断言

### 8.3 手动测试策略（D20 落地）

- **测试目标**：只验证 OS 边界（真实 Safari/Notes/Xcode 复制时 Bundle ID 识别是否正确）
- **测试场景**：
  - 在真实 Safari、Google Chrome、Trae IDE、VSCode、Xcode 中复制内容，验证白名单匹配
  - 配置保存目录为真实只读路径，验证文件写入失败处理
  - 模拟磁盘满场景，验证文件写入失败处理
- **证据存储**：手动测试记录存放在 `docs/planning/P1/F2.1/screenshots/` 与 `recordings/`

### 8.4 性能测试策略（D21 落地）

- **测试目标**：验证 NFR-001（≤3s）、NFR-002（≤500ms）、NFR-006（≤100ms）
- **测试方式**：记录实际耗时并断言 P95 满足约束，不使用固定 sleep 等待
- **测试场景**：
  - 1000 字内容保存延迟测试（NFR-001）
  - 剪贴板替换延迟测试（NFR-002）
  - F2.1 钩子额外延迟测试（NFR-006）
  - 连续复制 20 次的 P95 延迟测试（NFR-001）
- **性能测量工具**：`PerformanceMeasurer.measure(block:)` 记录耗时并计算 P95
