## 1. docs.md 10 项提取结果

**规则基准来源**：`.trae/rules/docs.md`

1. **需求文档章节列表**：docs.md 未显式提及"需求文档"，但 F2.1 父特性已采用 dd-write-requirements 默认 12 章节（Background / Problem Statement / Goals / Scope / Functional Requirements / Non-functional Requirements / Constraints / Acceptance Criteria / Out of Scope / Terminology / Decision Freedom / Future Considerations）。F2.1.1 沿用 12 章节。
2. **设计文档章节列表**：F2.1 父特性设计文档采用 dd-write-design 默认 10 章节（文档定位 / 模块划分 / 职责边界 / 数据流 / 状态变化 / 协作关系 / 关键设计决策 / 非功能约束落地 / 与 Requirements 的映射 / 风险和待确认问题）。F2.1.1 沿用 10 章节。
3. **文档编写顺序**：需求文档 → 设计文档 → 视觉原型（涉及 UI 时）→ 测试用例表。docs.md 原文为"设计规范 视觉原型 测试用例表"，F2.1 父特性补充了"需求文档"作为首篇。
4. **4 篇文档命名规则**：
   - `F2.1.1_保存成功Toast_需求文档.md`
   - `F2.1.1_保存成功Toast_设计文档.md`（注：docs.md 原文为"设计规范"，但 F2.1 父特性已使用"设计文档"命名，F2.1.1 子功能沿用父特性命名以保持局部一致性）
   - `F2.1.1_保存成功Toast_视觉原型.html`
   - `F2.1.1_保存成功Toast_测试用例表.md`
5. **文档存放路径**：`docs/planning/P1/F2.1/`（子功能文档挂在父功能目录下，与 F1.8 子功能模式一致）
6. **版本号和最后更新日期**：文档头部 `> 最后更新：YYYY-MM-DD | 版本：vX.Y`；测试用例表头部需注明 `> 最后更新：YYYY-MM-DD | 版本：vX.Y（基于需求文档 vA.B + 设计文档 vC.D）`
7. **其他文档要求**：视觉原型（涉及 UI 时必须，本特性涉及 UI 必须产出）、测试用例表（必须）
8. **标点符号规则**：中文标点（，。！？：；），英文术语保持原文
9. **mermaid 规则**：格式兼容 9.1.2，不使用中文标点、符号
10. **同步更新规则**：
    - 修改设计文档中的目标、范围、流程、接口、验收标准时，必须同步更新测试用例表
    - 涉及 UI 行为变化时，必须同步更新视觉原型
    - 每次关键变更必须追加版本记录和 history 日志（`docs/planning/P1/F2.1/historys/YYYY-MM-DD-修改摘要.md`）

## 3. 既有规格文档的可复用写作模式

**最近参考**：`docs/planning/P1/F2.1/F2.1_自动保存到文件_需求文档.md`（v1.2）与 `F2.1_自动保存到文件_设计文档.md`（v1.1）

**可复用模式**：
- 需求文档头部包含：功能编号、优先级、文档存放路径、适用阶段、前置依赖
- 需求文档使用编号引用：FR-001 起、NFR-001 起、AC-01 起、C-01 起（约束）、O-01 起（明确不做）、FC-01 起（未来扩展）
- 设计文档使用编号引用需求文档：FR-xxx / NFR-xxx / AC-xxx
- 设计文档第 9 章"与 Requirements 的映射"必须包含 FR 到模块的映射表
- 测试用例表对照代码中已有测试标注覆盖状态：✅ COVERED / 🟡 PARTIAL / ❌ MISSING / ⏸️ DEFERRED
- AC 写法：场景 + 预期 + 验证方式（必须有 XCTest 或 XCUITest）
