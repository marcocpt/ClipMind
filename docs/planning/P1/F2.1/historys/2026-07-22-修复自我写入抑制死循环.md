# 修复自我写入抑制死循环 bug

## 1. Bug 现象

开启自动保存功能后，设置长度阈值 50 字，在白名单 App（如 Safari）中复制 ≥50 字的长内容时，ClipMind 会不停地复制自己生成的文件路径，形成死循环：

1. 用户复制 ≥50 字内容 → F2.1 保存为文件 → 剪贴板替换为文件路径
2. PasteboardWatcher 检测到 changeCount 变化 → 文件路径被当作新复制内容
3. 文件路径长度 ≥50 字（路径含用户名、目录、文件名）→ 再次触发自动保存
4. 又保存一个文件，剪贴板再次被替换为新文件路径 → 回到步骤 2，死循环

## 2. 根因分析（systematic-debugging）

### 2.1 数据流追踪

按 FR-015/D4 决策，自我写入抑制机制应为：
- ClipboardReplacer 替换剪贴板为文件路径后调用 `markSelfWrite(newChangeCount)`
- PasteboardWatcher 下次轮询检测到 changeCount 变化时，调用 `checkAndReset(currentChangeCount)` 检查
- 命中标记（且未超时 5s）→ 静默跳过完整捕获流程，重置标记

### 2.2 断裂点定位

通过 Grep 搜索 `checkAndReset` 的所有调用点，发现：

| 调用位置 | 角色 | 是否调用 |
|---------|------|---------|
| ClipboardReplacer.swift:37 | 生产代码 - markSelfWrite | ✅ 已调用 |
| PasteboardWatcher.handlePasteboardChange | **生产代码 - checkAndReset** | ❌ **从未调用** |
| 各测试文件 | 测试代码 - checkAndReset | ✅ 已调用 |

**根因**：`SelfWriteSuppressor.checkAndReset` 只在测试代码中被调用，生产代码从未调用。装配代码（ClipMindApp.swift:236 `PasteboardWatcher(eventBuilder: eventBuilder)`）也没有把 suppressor 注入到 PasteboardWatcher。

自我写入抑制机制只实现了一半：标记了（markSelfWrite），但没人检查（checkAndReset）。

### 2.3 为什么 Deduplicator 没有拦截

Deduplicator 通过内容哈希去重。每次自动保存生成的文件路径都不同（文件名含内容前缀 + 冲突序号递增），所以 `isDuplicate` 始终返回 false，无法拦截死循环。

## 3. 修复方案

### 3.1 PasteboardWatcher 修改

1. 新增 `private let suppressor: SelfWriteSuppressor?` 属性
2. init 新增 `suppressor: SelfWriteSuppressor? = nil` 参数（默认 nil，F1.x 兼容）
3. `handlePasteboardChange` 在 `lastChangeCount = current` 之后、`contentReader.readContent` 之前，加 checkAndReset 检查：命中则记日志并 return，跳过完整捕获流程

### 3.2 装配修改

ClipMindApp.swift:236 `PasteboardWatcher(eventBuilder: eventBuilder)` 改为 `PasteboardWatcher(eventBuilder: eventBuilder, suppressor: suppressor)`，把已创建的 suppressor 注入到 PasteboardWatcher。

### 3.3 设计决策

- checkAndReset 检查位置：在 changeCount 变化检测后立即检查（最早时机），避免读内容、去重、构造事件等无效工作
- suppressor 默认 nil：保证 F1.x 既有测试兼容（不传 suppressor 时行为不变）
- 一次性标记：checkAndReset 内部会重置标记，避免重复抑制后续正常事件
