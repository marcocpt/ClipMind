# Docs 文档编写要求

`docs/` 下所有文档的命名、结构、编写流程和通用规范。

## 目录结构

- planning/
  - P{n}/F{m}/
    - historys/ 每次涉及 docs/、功能设计或关键架构决策的 commit 必须写日志。，命名 `YYYY-MM-DD-修改摘要.md`，内容：改了什么、为什么、影响哪些部分。
      `F{m}_{功能名}_设计规范.md`，必须，不包含具体代码设计，章节：1. 文档概述与项目背景（做什么/不做什么）2. 用户角色与业务流程（mermaid）3. 功能需求详述 4. 数据需求与接口定义 5. 非功能性需求 6. 验收标准（场景+预期+验证方式 必须有XCTest或XCUITest）。如需给 AI 自主编程补约束，设计规范只保留与需求、兼容性和完成定义直接相关的高层约束。
      `F{m}_{功能名}_视觉原型.html`，按需（涉及 UI 时），浏览器直接打开，编写时调用 `brainstorming` skill
      `F{m}_{功能名}_测试用例表.md`，必须，基于设计规范验收标准按功能分类生成测试用例矩阵。编写时调用 `brainstorming` skill，对照代码中已有测试标注覆盖状态（✅ COVERED / 🟡 PARTIAL / ❌ MISSING / ⏸️ DEFERRED）。文档头部版本号须注明对应的设计规范版本，格式：`> 最后更新：YYYY-MM-DD | 版本：vX.Y（基于设计规范 vA.B）`。

编写顺序：设计规范 视觉原型 测试用例表

## 通用要求

- `.md`
  - 标题后第一行：`> 最后更新：YYYY-MM-DD | 版本：vX.Y`，每次修改同步更新
  - 最后添加列表格式版本记录
- `.html` 页面头部显示最后更新时间与版本信息
- 链接用相对路径，不硬编码绝对路径
- 中文标点（，。！？：；），英文术语保持原文
- mermaid 用于流程图、状态图、类图
  - 格式兼容 9.1.2
  - 不使用中文标点、符号
- 没明确要求的都调用 `brainstorming` skill 编写
- 编码规范参考 `docs/CODING_STANDARDS.md`
同步更新规则：
  - 修改设计规范中的目标、范围、流程、接口、验收标准时，必须同步更新测试用例表。
  - 涉及 UI 行为变化时，必须同步更新视觉原型。
  - 每次关键变更必须追加版本记录和 history 日志。
